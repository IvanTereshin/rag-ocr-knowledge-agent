# План реализации: RAG OCR Knowledge Agent

## 1. Цель проекта

Создать AI-агента, который отвечает на вопросы по базе знаний компании. База знаний собирается из документов разных форматов:

- `PDF`
- `PPT/PPTX`
- `DOC/DOCX`
- `TXT/MD`
- `CSV`
- `XLS/XLSX`
- изображения и сканы

В проекте должны быть два режима:

- Облачный (`portfolio-top`) - лучший стек для портфолио, с внешними API и высоким качеством.
- Локальный (`local-self-host`) - локальный стек, который можно запустить без внешних AI API.

## 2. MVP-результат

Пользователь должен уметь:

1. Создать workspace.
2. Загрузить документы.
3. Увидеть статус обработки каждого документа.
4. Задать вопрос по базе знаний.
5. Получить ответ с источниками.
6. Переключить RAG-профиль.
7. Сравнить ответ режимов Облачный и Локальный на одном вопросе.

## 3. Рекомендуемый стек приложения

### Frontend

- Next.js или React + Vite.
- TypeScript.
- UI: shadcn/ui или Telegram-like lightweight UI, если потом нужен TMA.
- TanStack Query для API-запросов.

### Backend

- Node.js + NestJS или Fastify.
- PostgreSQL для пользователей, документов, задач обработки.
- Qdrant для vector search.
- Redis + BullMQ для очередей обработки документов.
- S3-compatible storage или локальный `uploads/` для MVP.

### DevOps

- Docker Compose.
- Отдельные сервисы:
  - `web`
  - `api`
  - `worker`
  - `postgres`
  - `redis`
  - `qdrant`
  - `ollama` для local self-host

## 5. Главные сущности

### Workspace

База знаний компании или отдела.

Поля:

- `id`
- `name`
- `created_at`

### Document

Загруженный файл.

Поля:

- `id`
- `workspace_id`
- `file_name`
- `file_type`
- `size_bytes`
- `status`
- `processing_profile`
- `error_message`
- `created_at`

### DocumentChunk

Кусок документа, который участвует в поиске.

Поля:

- `id`
- `document_id`
- `text`
- `metadata`
- `token_count`
- `vector_id`

### QuestionRun

Один запуск вопроса.

Поля:

- `id`
- `workspace_id`
- `question`
- `profile`
- `answer`
- `citations`
- `latency_ms`
- `cost_estimate`
- `created_at`

## 6. Профиль Облачный (`portfolio-top`)

### Цель

Показать качество и production-подход.

### Компоненты

- Парсинг: Docling или Unstructured.
- OCR: Mistral OCR 4.
- Vector DB: Qdrant.
- Embeddings: OpenAI embeddings.
- Rerank: Cohere Rerank или BGE Reranker.
- Генерация ответа: OpenAI Responses API.

### Что обязательно показать

- OCR сложного PDF.
- Таблица из Excel.
- Слайд из PPTX.
- Ответ с несколькими источниками.
- Сравнение с локальным режимом.

## 7. Профиль `local-self-host`

### Цель

Показать приватный запуск без внешних AI API.

### Компоненты

- Парсинг: Docling.
- OCR: PaddleOCR, fallback Tesseract.
- Vector DB: Qdrant local.
- Embeddings: BGE-M3.
- Rerank: BGE Reranker v2 m3.
- Генерация ответа: Ollama или vLLM.

### Что обязательно показать

- Индексация без внешних API.
- Ответ по локальным документам.
- Логи использования локальных моделей.
- Замер latency.
- Пояснение, какие данные не уходят наружу.

## 8. Этапы разработки

### Этап 1. Каркас проекта

Результат: приложение запускается локально.

Задачи:

- Создать monorepo.
- Настроить frontend, backend и worker.
- Поднять PostgreSQL, Redis, Qdrant через Docker Compose.
- Добавить `.env.example`.
- Добавить `RAG_PROFILE`.
- Сделать healthcheck API.

### Этап 2. Загрузка документов

Результат: пользователь может загрузить файл, а backend создает запись `Document`.

Задачи:

- Сделать экран загрузки.
- Поддержать drag-and-drop.
- Ограничить типы файлов.
- Сохранять файл в storage.
- Создавать job в очереди.
- Показывать статус обработки.

### Этап 3. Базовый парсинг документов

Результат: документы превращаются в нормализованный текст и блоки.

Задачи:

- Подключить Docling.
- Обработать `PDF`, `DOCX`, `PPTX`, `TXT`, `XLSX`.
- Для Excel сохранить листы, строки, колонки.
- Для PPTX сохранить номер слайда.
- Для PDF сохранить номер страницы.
- Сохранять результат в PostgreSQL.

### Этап 4. OCR-слой

Результат: сканы и изображения тоже попадают в базу знаний.

Задачи для `portfolio-top`:

- Подключить Mistral OCR 4.
- Сохранять bounding boxes и confidence, если доступны.
- Добавить OCR fallback для PDF с низкой плотностью текста.

Задачи для `local-self-host`:

- Подключить PaddleOCR.
- Добавить fallback на Tesseract.
- Сохранять OCR-результат в том же формате, что и cloud OCR.

### Этап 5. Chunking

Результат: документы режутся на удобные куски для поиска.

Задачи:

- Сделать layout-aware chunking.
- Не разрывать таблицы без необходимости.
- Не смешивать разные слайды PPTX в один chunk.
- Для Excel делать chunks по листам/таблицам.
- Добавлять metadata: файл, страница, слайд, лист, таблица, строки.

### Этап 6. Embeddings и индекс

Результат: chunks можно искать.

Задачи:

- Реализовать общий интерфейс `EmbeddingProvider`.
- Для `portfolio-top` подключить OpenAI embeddings.
- Для `local-self-host` подключить BGE-M3.
- Сохранять vectors в Qdrant.
- Сохранять metadata в payload.

### Этап 7. Hybrid search

Результат: поиск работает и по смыслу, и по точным словам.

Задачи:

- Dense vector search.
- Keyword/sparse search.
- Объединение результатов.
- Настраиваемые веса.
- Логирование найденных chunks.

### Этап 8. Reranking

Результат: в LLM попадают лучшие источники.

Задачи:

- Реализовать общий интерфейс `Reranker`.
- Для Облачного (`portfolio-top`) подключить Cohere, Voyage AI или Jina AI.
- Для Локального (`local-self-host`) подключить TEI + `BAAI/bge-reranker-base`.
- Ранжировать top 30 в top 5-8.

### Этап 9. Генерация ответа

Результат: пользователь получает ответ с источниками.

Задачи:

- Реализовать общий интерфейс `AnswerGenerator`.
- Для Облачного (`portfolio-top`) подключить OpenAI Responses API.
- Для Локального (`local-self-host`) подключить Ollama/vLLM.
- Запретить отвечать без источников, если вопрос требует фактов.
- Добавить формат ответа:
  - короткий ответ;
  - детали;
  - источники.

### Этап 10. Citations

Результат: ответ можно проверить.

Задачи:

- Для PDF показывать страницу.
- Для PPTX показывать слайд.
- Для Excel показывать лист и диапазон строк.
- Для DOCX показывать секцию/заголовок, если есть.
- В UI сделать кликабельные источники.

### Этап 11. Переключение режимов

Результат: пользователь может выбрать режим.

Задачи:

- Глобальный default profile через `.env`.
- Переключатель в UI.
- Возможность переиндексировать документ под другим профилем.
- Хранить `processing_profile` у документа.
- Хранить `profile` у каждого вопроса.

### Этап 12. Экран сравнения

Результат: сильная демо-фича для портфолио.

Задачи:

- Один вопрос запускается в двух режимах.
- UI показывает два ответа рядом.
- Показать:
  - latency;
  - найденные источники;
  - confidence/score;
  - примерную стоимость;
  - локальные ресурсы для self-host.

### Этап 13. Evals

Результат: есть простая проверка качества.

Задачи:

- Создать набор тестовых документов.
- Создать 20-30 контрольных вопросов.
- Для каждого вопроса указать правильный источник.
- Проверять retrieval hit rate.
- Проверять наличие citations.
- Сохранять отчет по двум режимам.

### Этап 14. Demo pack

Результат: проект легко показать клиенту или в портфолио.

Задачи:

- Подготовить демо-документы:
  - policy PDF;
  - sales deck PPTX;
  - pricing XLSX;
  - onboarding DOCX;
  - scanned invoice PDF.
- Подготовить 10 демо-вопросов.
- Сделать README со скриншотами.
- Записать короткий demo-flow.

## 11. Риски

### Риск: OCR ошибается на плохих сканах

Решение:

- показывать confidence;
- давать пользователю увидеть OCR-текст;
- разрешить переобработку другим OCR-профилем.

### Риск: LLM придумывает ответ

Решение:

- отвечать только по найденным источникам;
- показывать источники;
- если источников нет, честно писать "в базе знаний не найдено".

### Риск: Excel плохо превращается в текст

Решение:

- хранить лист, таблицу, строки, колонки;
- делать отдельные chunks по таблицам;
- не смешивать разные листы.

### Риск: локальный режим медленный

Решение:

- кешировать embeddings;
- использовать меньшую модель для демо;
- показывать latency как честную метрику.

## 12. Definition of Done

Проект можно считать готовым, когда:

- загружаются минимум `PDF`, `DOCX`, `PPTX`, `XLSX`, `TXT`;
- сканированный PDF проходит OCR;
- оба режима умеют индексировать документы;
- оба режима умеют отвечать на вопросы;
- ответы содержат источники;
- есть экран сравнения двух режимов;
- есть демо-документы и демо-вопросы;
- проект запускается одной командой через Docker Compose.
